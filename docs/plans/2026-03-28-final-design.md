# DocSpec Final Design — Self-Documenting System

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Build a fully working self-documenting system where DocSpec documents and tests itself, starting with Java, then replicating to other languages.

**Visual Reference:** `docspec-poc.jsx` in repo root — this is the design system source of truth.

---

## Decision 1: Two Lenses, Not Three

**Docs** and **Tests** are the two lenses. Changelog is a page type within Docs (under Architecture).

```
[Docs] [Tests]
```

**Critical UX rule:** When you switch lenses, the sidebar selection STAYS THE SAME. If you're looking at `IntentGraphExtractor` in Docs and switch to Tests, you see `IntentGraphExtractor`'s test view. Same sidebar position. No navigation. No context loss.

## Decision 3: What Each Page Shows Per Lens

### OVERVIEW / INTRODUCTION

**Docs lens:**
- Project cards with version, class count, description
- Architecture-at-a-glance diagram
- Quick stats: classes, methods, flows, endpoints
- Cross-project connections summary
- "Meta-dogfooding · DocSpec documents itself" badge

**Tests lens — Dashboard:**
- Stats cards: total tests, coverage %, avg ISD, gap count
- Class Health Treemap (≥30 classes) or Table (<30 classes): every class sized by method count, colored by ISD (green >8, yellow 4-8, red <4). Gap classes have pulsing yellow border. Click → class tests.
- Channel breakdown: horizontal bars with test count + bugs found
- Cross-channel gaps: severity-sorted list
- ISD distribution histogram

## Decision 4: Cross-Project

Only appears when `artifacts.length > 1` AND crossRefs/DocUses exist.

- Project Switcher in sidebar header
- "USED BY" section on the project being called into
- "DEPENDS ON" section on the calling project
- Dependency graph zooms: project-level → class-level
- Data Stores show cross-project table ownership
- Changelog shows CROSS-PROJECT IMPACT sections

## Decision 5: Edge Cases

| Case | Behavior |
|------|----------|
| ISD = 0 | Method shows `⚪ 0t ISD 1.0` + tooltip: "Low intent density" |
| @DocTestSkip | Grayed: `legacyProcess — SKIP "Scheduled for removal"` |
| @DocIntentional | Gap acknowledged: `✓ Acknowledged — "includes protected by design"` |
| Verified example broken | `❌ Example broken — Last passed in v2.9.0` |
| Long method (400+ lines) | Source View shows first 100 lines + "Show full source" |
| Circular dependency | Red bidirectional edge with tooltip |
| Property test counterexample | Shows shrunk counterexample values |
| Empty module | Dimmed: `📦 internal (0 classes)` |

## Decision 6: Every Badge Is A Link

- `🧪 19t` on method → Tests lens, same class, that method expanded
- `ISD 12.5` → Tests lens, ISD breakdown visible
- `⚠ 1 gap` → Tests lens, scrolled to gap
- `Phase 4` on test → Docs lens, Flow page, step 4 highlighted
- `C` class badge → Docs lens, class page
- Channel tag `🛡️ Guards` → Tests lens, filtered to Guard channel
- Project dot `●` → switch project context

## Decision 9: Full Visibility Scanning + Auth-Based Access Control

**Scan everything.** The processor discovers ALL classes, methods, fields regardless of visibility (public/protected/private/package). Each element is tagged with its actual visibility level and `@DocAudience` annotation.

**Authentication controls who sees what:**
- `@DocAudience("public")` → visible to everyone (API reference)
- `@DocAudience("partner")` → authenticated partners (SDK details)
- `@DocAudience("internal")` → team only (architecture, private methods, DSTI, full dependency graph)

**Auth providers (pluggable via CLI):**
- `npx docspec auth setup` — interactive wizard (@clack/prompts)
- Supported: Google OAuth, GitHub OAuth, Microsoft/Azure AD, Kerberos/LDAP, OIDC, custom plugin
- Config saved to docspec.config.yaml
- `npx docspec dev` starts with auth middleware when configured
- `npx docspec dev --auth none` for development (show everything)

**Why:** Private fields ARE the dependency graph. Private methods ARE what DSTI tests. Hiding them defeats the purpose. Instead, control access — internal team sees everything, external users see only public API.

## Decision 10: @DocMethod Replaces JavaDoc

DocSpec annotations are the single source of truth for documentation. No `/** */` JavaDoc needed.

- `@DocMethod(value = "description", params = {@Param(...)}, returns = "...", throwsDoc = {@ThrowsDoc(...)})` replaces all JavaDoc tags
- `@DocBoundary("description")` replaces class-level JavaDoc
- `docspec:javadoc` Maven goal generates standard JavaDoc HTML from docspec.json (Maven Central compatibility)
- `docspec:inject-sources` generates source copies with `/** */` injected (IDE hover compatibility)

## Implementation Order — Java First

### Phase 1: Annotate Java Codebase
1. Add `@DocFlow` for the processing pipeline (top-level + sub-flows)
2. Add `@DocError` on all exception conditions
3. Add `@DocEvent` on processing events
4. Add `@DocContext` on Maven Mojos
5. Add `@DocUses` on cross-module dependencies
6. Add `@DocExample` on key methods
7. Add `@DocPerformance` on hot paths
8. Add `@DocInvariant` with Property DSL rules
9. Enrich `@DocMethod` descriptions on remaining methods
10. Add `@DocBoundary`, `@DocDeterministic`, `@DocIdempotent`, `@DocCommutative`

### Phase 2: Fix the Processor
1. Fix ISD scores (currently all zeros)
2. Verify all 13 channels produce signals
3. Run cross-channel verifier, confirm gaps detected
4. Generate docspec.json with flows, errors, events, models

### Phase 3: Rewrite the Site Theme
1. Replace Stripe light theme with dark theme from POC
2. Implement two-lens architecture (Docs/Tests toggle)
3. Implement unified sidebar with lens-aware badges
4. Implement class page (Docs lens) matching POC ClassPage
5. Implement class page (Tests lens) with Source View + Card View
6. Implement test dashboard with treemap/table
7. Implement flow page with project boundaries
8. Implement dependency graph (interactive SVG)
9. Fix data loading pipeline end-to-end
10. Fix routing for all page types

### Phase 4: End-to-End Verification
1. Run processor on DocSpec's own code
2. Run CLI resolve → verify 140+ pages
3. Start dev server → verify every page type renders
4. Click through every link type → verify navigation
5. Switch lenses on every page → verify context preserved
6. Verify gaps are detected and displayed in both lenses

### Phase 5: Replicate to Other Languages

Each language follows the same pattern: annotate → run processor → verify docspec.json → verify site renders.
